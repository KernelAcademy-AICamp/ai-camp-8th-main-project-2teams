# 맥락 (Context)
> 유형: foundation · 2026-07-21 · 방법: 0단계 맥락 덤프 + 라우팅

## 원 상황 (raw)
부트캠프 메인 프로젝트로 "AI 쇼핑 발견 도우미" 원안(구매의도 기반 추천 / AI 추천 준비도 진단 / 리뷰 리스크 요약, 30~50개 카탈로그)이 주어졌으나, 우리 팀은 이를 **프린팅 반팔 티셔츠 카테고리 + LLM 검색**으로 압축해 진행하기로 함.

목표: "프린팅 반팔 티셔츠로 압축한 상태에서 LLM 검색을 통해 원하는 상품을 찾는다." 네이버 오픈API를 우선 사용해 검토. 6주, 주 단위 스프린트, 4명(개발자는 본인 1명, 나머지 비개발).

## 되물어 파악한 것
| 질문 | 답 | 해석 |
|---|---|---|
| 핵심 문제(LLM 검색으로 노리는 지점)? | "노란 프린팅 흰티를 찾을 검색어·필터가 없음 + 산들산들한 느낌을 성분·댓글로 검색하고 싶음" | 문제 = **말로 표현한 시각 속성(색×프린팅 조합) + 감각 속성(촉감·느낌)** 을 못 찾음. 키워드/필터의 한계. |
| 성분·리뷰 데이터를 어떻게 채우나? | "이미지 ML로 색 정보, OCR로 이미지 속 텍스트 추출 등 방법 조사→자동화→수동 샘플검증→정확도 판단→신뢰도 확보 후 2000개로 1차" | **자동 추출 + 정확도 검증 루프**. 비개발 3명이 검증·라벨링 담당하기에 적합. v1 소규모→확장. |
| v1 검색축? | "시각 + 성분/리뷰 둘 다" | 야심적. 6주/1인 위험 → 하나의 속성 테이블로 통일하고 신뢰도 순으로 배치(아래 진입점·D5 참고). |
| 성분·리뷰 소스? | "네이버 스마트스토어 생태계로 통일" | 상세·리뷰가 한 생태계. 단 조사 결과 소재는 정형 필드 없음·리뷰는 공식 API 없음(data-feasibility). |
| 리뷰 v1 처리? | "소규모 샘플 실험만" | 리뷰 크롤링을 핵심 의존축에서 제외. 가능성만 검증하고 v2 확장 근거로. |

## 초기 메타인지 플래그
- **솔루션 바이어스 주의**: 기능 골격(원안의 3기능)이 부트캠프에서 이미 주어져 "무엇을 만들지"가 정해진 상태로 출발. → "사용자가 정말 이렇게 찾는가"(문제 진위)는 아직 미검증. **다음 단계에서 반드시 실제 사용자 목소리로 검증** 필요.
- **기술 주도 스코프**: 데이터 파이프라인(이미지ML·OCR·크롤링) 이야기가 먼저 깊어짐. 정작 "누가·왜 이렇게 티셔츠를 찾는가"(타겟·JTBD)는 미확정. 다음 단계에서 채운다.
- **1-Way 결정 후보**: 데이터 스키마(속성 테이블 구조), 데이터 소스(네이버 의존) — 나중에 바꾸기 비쌈. 스키마 확정 전 검색축 합의 필요(D5로 1차 정리).

## 진입점 결정
- 가장 불확실한 것: **① 문제가 진짜인지(사람들이 정말 이 방식으로 티셔츠를 찾나) + ② 데이터로 이게 되는지(feasibility)**
- 라우트: **A(문제 진위) + C(데이터/도메인 feasibility) 혼합.** feasibility는 data-feasibility.md에서 1차 조사 완료 → 다음은 **문제 진위 + 타겟 세그먼트**로 진행.
